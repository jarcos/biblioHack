Bounded contexts (DDD + hexagonal)

One hexagon per context, each with domain / application / infrastructure / interfaces layers under backend/src/bibliohack/. The domain never imports infrastructure; every concrete dependency enters through a port and is injected via FastAPI Depends.

Stack decisions that stuck (and the two that didn't)

  • Postgres as the single store — relational + vectors (pgvector) + time-series (Timescale) + FTS. Qdrant remains a swap-in behind the Embedder port if p95 > 250 ms.
  • Scrapling / Patchright Chromium for the JS-rendered OPAC; the "lightweight httpx path" did not survive contact — the OPAC is a JS SPA with session-token redirects.
  • Astro + React islands, TanStack Query, Tailwind + shadcn/ui. Vitest + fast-check on domain/application TS.
  • Deviation 1 — embeddings are not local. BGE-M3 via the HuggingFace Inference API; the torch model didn't fit the NAS RAM next to the monitoring stack. Same vectors, same schema; a local adapter stays possible behind the port.
  • Deviation 2 — no event bus (yet). The planned in-process bus (recommendation invalidation on BookRead/BookRated) was replaced by a shelf-fingerprint cache key (SHA-256), which is cross-process-safe — imports happen in the worker process, so an in-process bus would never have reached the api anyway.

The polite crawler

Compliance is a feature. The Junta's terms allow reuse (CC-BY, attribution required) but forbid overloading the service — so the politeness budget is a hard requirement: 1 request/s jittered, single-threaded, nightly window, 30k/day cap, exponential backoff, identified User-Agent. Discovery enumerates the TITN space (high-water mark 2,662,739 probed in 42 fetches); ingest parses rendered HTML (no MARC export exists); refresh is tiered by hotness with source_hash short-circuiting unchanged records. The crawler ships as an on-NAS container deliberately outside the prod compose so its ~3.5 GB image build can never block api/frontend CD.

Deployment & operations

  • Topology: Synology NAS (x86-64), Docker Compose, published through the pre-existing Cloudflare Tunnel. No inbound ports; TLS at the edge. Postgres binds LAN-only.
  • Routing rule that bites: one origin; the tunnel sends /api/*, /catalog/*, /healthz to the api and everything else to the static frontend — any new frontend-called endpoint must live under /api/* or it returns page HTML.
  • Services: postgres, api, frontend, minio, redis (since identity), plus the crawler and the Dramatiq import worker as separate compose projects.
  • CI/CD: green push to main → backend gate (ruff, mypy, pytest w/ testcontainers) + frontend gate (prettier, eslint, astro check, vitest) + image builds → auto-deploy to the NAS. Red pipeline never deploys.
  • Observability: api auto-instrumented with OpenTelemetry (FastAPI + asyncpg) → OTLP → shared collector → Grafana Tempo + SigNoz. No-op without OTEL_* env vars, so dev/test are unaffected. Crawl/worker plane not yet traced — health comes from scrape_tasks / import_jobs status rows.
  • Mail: Mailgun EU (smtp.eu.mailgun.org:587) behind the Mailer port — a deviation from the planned NAS SMTP relay.
  • Backups: nightly pg_dump + Hyper Backup of /volume1/docker/bibliohack.

Security posture

The tunnel exposes the frontend and the /api/* surface; MinIO console, Postgres and worker controls stay on LAN/Tailscale. Sessions are opaque 256-bit ids in Redis behind an httpOnly, Secure, SameSite=Lax cookie — trivially revocable, no JWT rotation pain. Registration and login are Turnstile-gated and rate-limited (fail-open by design: Redis being down must not lock everyone out of login, which wouldn't work without Redis anyway). Account deletion requires the password again — a stolen cookie alone can't destroy data. The public app only ever reads our mirror; it never triggers a live OPAC call per request.
