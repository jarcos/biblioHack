User Accounts — Action Plan

biblioHack: from single-reader homelab to multi-user app with public registration

Source: Cowork session of 2026‑06‑09 · full plan in docs/identity-milestone-plan.md Status: Phases 0 + 1 kicked off together (José's last instruction)

Why this exists

José flagged that the app is single-user and asked for user accounts: register/login with each user persisting their own bookshelf. Investigation confirmed single-user is a deliberate scoping decision, not an oversight — shelf.py says "there is exactly one reader," and an empty identity package plus ARCHITECTURE.md §4 already spec the Identity context (User root, UserRepository/AuthProvider ports, Argon2id). The work is implementing it and retrofitting everything currently single-tenant.

Verified starting point: no auth anywhere; shelf_entries is globally keyed with no user_id; shelf imports are CLI-only (public users have no upload path); the recommendations context is empty scaffolding — so it gets built user-scoped from day one rather than retrofitted.

Decisions locked

TopicDecisionNotes
Registration model Decided Public registration Anyone can sign up — drives the abuse-protection and GDPR work. registration_enabled kill-switch in settings.
Recommendations Decided Scoped by user_id Greenfield build; per-user profile, cache, and event-driven invalidation.
Existing shelf data Decided Discard & re-import José doesn't mind losing current entries (CSV re-import). Migration becomes trivial: truncate, add user_id NOT NULL on the empty table — no backfill, no owner bootstrap. The riskiest step of the original plan is gone.
Email verification Decided Yes, via NAS SMTP The home NAS already has a mailing service connected; the Mailer adapter points at it. No new provider.
CSV import execution Decided Dramatiq background worker Endpoint returns 202 + import id; worker processes the job; frontend polls status. See rationale below.
GDPR posture Decided Act as data controller Ship /privacy and /terms pages, signup consent checkbox, account deletion (cascade), and data export.
Session mechanism Recommended Redis sessions + httpOnly cookie Over stateless JWT: Redis already in stack, trivially revocable, no secret-rotation pain, same-origin behind the tunnel.
Bot protection Recommended Cloudflare Turnstile On register/login. Already fronted by Cloudflare; free, no CAPTCHA UX hit.
Sync vs Dramatiq — why background imports won. Each CSV row fires two trigram queries against the ~1.5M-record catalogue. Inline imports block the browser, risk tunnel/HTTP timeouts on large Goodreads libraries, hold an asyncpg connection for the duration, and concurrent public uploads can saturate the pool and degrade catalogue search for everyone. The Dramatiq worker (Redis broker, already in the stack; container alongside bibliohack-crawler) bounds DB load by worker count, retries transient failures, and survives client disconnects. ARCHITECTURE.md already names Dramatiq + Redis (§5.1) and the ImportJob aggregate (§4) — it's the intended design, not an add-on.

The plan — six phases

0

Schema & settings groundwork In progress

Database and config foundations; app still works single-user after this ships.

  • users table: id UUID PK, email CITEXT UNIQUE, password_hash (Argon2id), email_verified, display_name, timestamps. Companion tables for email-verification and password-reset tokens (storing token hashes only).
  • shelf_entries retrofit (simplified): truncate → add user_id UUID NOT NULL → FK to users with ON DELETE CASCADE → swap unique constraint to (user_id, source, source_book_id) → index on user_id. One Alembic revision, runs on deploy.
  • Settings: session secret/TTL/cookie config, Argon2 tuning, registration_enabled kill-switch, NAS SMTP config, Turnstile keys.
1

Identity context — auth In progress

Full hexagonal context mirroring the others; no scoping of existing features yet.

  • Domain: User aggregate, Email/PasswordHash value objects, Preference reserved for later.
  • Ports: UserRepository, PasswordHasher, SessionStore, TokenService, Clock, Mailer.
  • Use cases: RegisterUser, AuthenticateUser, VerifyEmail, RequestPasswordReset, ResetPassword, GetCurrentUser — tested against in-memory fakes.
  • Adapters: Postgres repo, argon2-cffi hasher, Redis session store, SMTP mailer → NAS service.
  • Endpoints under /api/auth/* (must stay under /api/* per tunnel routing): register, verify, login, logout, me, password reset-request/reset. Register + login Turnstile-gated; login rate-limited.
  • Cross-cutting dependencies: get_current_user (401 on missing/expired session) and get_optional_user (catalog search stays public).
2

Scope ReadingHistory by user

Every shelf read/write carries a user_id; web users get an upload path.

  • Thread user_id through ShelfEntryData, ShelfRepository, read repositories, and ImportShelf; GET /api/shelf requires auth.
  • New POST /api/shelf/import: multipart CSV upload → Dramatiq job → 202 + import id, with max-upload-size and row-cap guardrails.
  • ImportJob aggregate + import_jobs table (status, per-shelf stats, error) and GET /api/shelf/import/{id} for the polling UI.
  • CLI stays for the owner, gaining --user-email to resolve a user instead of assuming the sole reader.
3

Frontend auth (Astro + React islands)

The user-facing surface: forms, guards, and legal pages.

  • Pages: /register, /login, /account, /verify, password-reset; island forms call /api/auth/* with credentials: "include".
  • Astro middleware reads the session cookie server-side; /shelf and /recommendations redirect to /login when unauthenticated.
  • Account page: change password, export my data, delete my account (GDPR).
  • /privacy and /terms pages, linked from footer and registration form, with an enforced consent checkbox at signup.
  • Turnstile widget on register/login.
4

Recommendations — user-scoped from the start

Greenfield build of the empty context per ARCHITECTURE.md §4.

  • recommendations table keyed by user_id (FK, cascade delete) with score, rationale, per-user cache key.
  • Engine builds a per-user profile from that user's shelf (pgvector centroid / liked-item retrieval), runs hybrid FTS + pgvector retrieval over the shared catalogue, excludes books already on the user's shelf, asks OpenRouter for rationales.
  • BookRead/BookRated events carry user_id; recommender invalidates that user's cache.
  • GET /api/recommendations requires auth and never returns another user's data.
5

Hardening for public exposure

The price of public registration on a home NAS.

  • Rate limiting on /api/auth/* and the import endpoint (slowapi or Cloudflare edge).
  • Per-user isolation tests (testcontainers): user A can never read or write user B's shelf or recommendations; schemathesis contract tests on new endpoints.
  • Account deletion cascades through shelf and recommendations; sessions invalidated.
  • Data export endpoint; consent and privacy notice wired in.
  • OTel: new FastAPI/asyncpg paths get spans for free; add opentelemetry-instrumentation-redis if session-store spans are wanted. The Dramatiq worker is not instrumented — job health comes from import_jobs status rows.

Build & ship order

Each step independently shippable and green-gated (ruff format --check, ruff check, mypy src, pytest), then auto-deploys to the NAS from main:

  • 1. Phase 0 migration + settings → 2. Phase 1 Identity + auth endpoints → 3. Phase 2 shelf scoping + import endpoint → 4. Phase 3 frontend auth → 5. Phase 4 recommendations → 6. Phase 5 hardening.
Current status: José asked to start Phase 0 and Phase 1 together (last message of the session, 2026‑06‑09). After Phase 0 ships, the owner registers like any other user and re-imports the Goodreads CSV through the new flow.
