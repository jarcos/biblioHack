M1: SeedDiscoveredTasks use case + `bibliohack catalog seed` CLI

Second piece of M1 step 26: turn a TITN range into `discovered` rows.

Adds:
- catalog/application/use_cases/seed_discovered_tasks.py: pure orchestration
  on top of ScrapeTaskRepository.seed_range. Returns a SeedResult DTO with
  the range bounds, count of new rows inserted, and a derived `already_known`
  count. Validates low <= high.
- catalog/interfaces/cli.py: `bibliohack catalog seed --from N --to M`
  subcommand. Wires up an AsyncSession via the new composition helper,
  bulk-inserts under a single transaction, prints a coloured summary.
- shared/infrastructure/composition.py: transactional_session() async
  context manager so CLI commands and one-off workers can grab a DB
  session without re-implementing the engine/factory dance. Auto-commits
  on clean exit, rolls back on exception, disposes the engine either way.
- Makefile: `seed-titn HIGH=N` convenience target.
- 4 unit tests covering the use case with a fake repository.

Verified manually against the running docker-compose Postgres:

    bibliohack catalog seed --from 1 --to 10000          # 10000 inserted
    bibliohack catalog seed --from 5000 --to 15000       # 5000 new + 5001 already-known
    psql ... > 15000 rows total, all 'discovered'

Bug caught and fixed during the manual run: asyncpg caps prepared statements
at 32_767 arguments, NOT the 65_535 PostgreSQL itself allows. SQLAlchemy
emits 4 params per row (titn + Python-side defaults for status /
attempt_count / priority), so the original 50k-row chunk overflowed.
Reduced _INSERT_CHUNK_SIZE to 8_000 (8000 * 4 = 32_000 args, safe).

142 tests passing, coverage 89.83%, ruff/format/mypy clean.
